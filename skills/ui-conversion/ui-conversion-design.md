## Quando usar

Use em paginas de venda, VSLs, quizzes, advertoriais, diagnosticos e calculadoras para produtos low ticket.

## Principios

- Comece mobile-first; a maior parte do trafego pago e organico chega pelo celular.
- O primeiro viewport precisa explicar problema, promessa, mecanismo e proximo passo sem poluir.
- Cada bloco deve ter uma funcao de conversao: reconhecer dor, explicar mecanismo, reduzir risco, provar, mostrar oferta ou conduzir acao.
- Use contraste alto, tipografia simples, botoes claros e espacamento previsivel.
- Evite layout de landing page generica quando a oferta pede interacao.
- Nao use provas, selos, depoimentos ou resultados sem origem.
- Nao copie referencias; extraia padroes e redesenhe para a oferta.

## Componentes de conversao

- Hero com promessa especifica e CTA visivel.
- Diagnostico rapido com 3 a 6 perguntas.
- Blocos de mecanismo unico com visual simples.
- Cards de sintomas/erros para reconhecimento.
- Stack de oferta com produto principal, bonus, order bump e upsell quando aplicavel.
- Barra de progresso para quizzes e desafios.
- Comparativo "jeito comum vs mecanismo da oferta".
- FAQ focado em objecoes reais.
- Checkout handoff claro, sem surpresa.

## Regras de UI

- Botoes devem parecer clicaveis e ter texto de acao.
- Cards devem ser usados para itens repetidos, nao para toda a pagina.
- Evite paleta de uma cor so.
- Mantenha textos curtos dentro de componentes compactos.
- Elementos interativos precisam ter estado inicial, progresso, erro e conclusao.
- A versao desktop deve ampliar clareza, nao virar outra experiencia.
